BUSINESS RULES SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PARTIAL PAYMENTS:
  • Require customer for partial payments (track balance)
  • Cash payments: auto-verified
  • GCash payments: pending → admin verifies
  • Payment amount must be > 0 and ≤ balance_remaining
  • Can mix payment methods (cash + gcash + cash)

STAGGERED PLANS:
  • Require customer (cannot be anonymous)
  • Minimum down payment: 10% (configurable)
  • Maximum installments: 12 (configurable)
  • Late fees after grace period (3 days default)
  • Can pay installments early (no penalty)
  • Partial installment payments allowed

PDO:
  • Require customer + admin approval
  • Minimum order: ₱5,000 (configurable)
  • Maximum period: 90 days (configurable)
  • Check must be dated 1+ days in future
  • Maximum 2 date extensions
  • After 3 bounced checks: customer blocked
  • Credit limits based on payment history

  1. REVISED CONCEPT OVERVIEW
================================================================================

  OLD APPROACH (v1 — SCRAPPED):
    - Add a 5th payment method called "Installment"
    - Only installment orders could have partial payments

  NEW APPROACH (v2 — THIS DOCUMENT):
    - NO new payment method needed
    - ALL 4 existing payment methods support partial payments natively
    - A dedicated PAYMENTS PAGE manages, verifies, holds, or cancels payments

  How it works now — ALL methods support partial:
  -----------------------------------------------------------------------
    - Cash      → Enter amount paid → if amount < total → balance remains
    - GCash     → Enter amount paid → if amount < total → balance remains
    - COD       → Enter amount paid → if amount < total → balance remains
    - Pay Later → No payment now → full balance → pay later (enter amount,
                  if amount < total → balance still remains)
  -----------------------------------------------------------------------

  Example Flow:
    1. Customer orders ₱1,000 worth of goods
    2. Pays ₱500 via Cash → payment_status = 'partial', balance = ₱500
    3. Comes back, pays ₱300 via GCash → balance = ₱200
    4. Comes back, pays ₱200 via Cash → balance = ₱0 → payment_status = 'paid'

  Key Insight: Each subsequent payment can use a DIFFERENT method.
    - 1st payment: Cash ₱500
    - 2nd payment: GCash ₱300
    - 3rd payment: Cash ₱200
    → The sale's payment_method stays as original (cash), but each
      individual payment record has its own method.

  ★ NEW: PAYMENTS PAGE (Payment Verification & Management)
  -----------------------------------------------------------------------
  WHY: What if someone sends a GCash screenshot but walang pumasok?
       What if the reference number is wrong? What if they uploaded
       a fake proof of payment?

  The Payments page lets admin:
    - SEE all recorded payments across all orders
    - VERIFY — confirm a payment is legitimate (verified status)
    - HOLD — flag a suspicious payment for review (on_hold status)
    - CANCEL/REJECT — reject a bad payment, reverting the balance
      (e.g., GCash proof but no actual money received)
    - This ensures financial integrity — payments aren't just
      "fire and forget", they go through a verification flow

  Payment Status Flow:
    Cash payments:    → auto 'verified' (cash is immediate, no proof needed)
    GCash payments:   → starts as 'pending' → admin verifies → 'verified'
                      → OR admin flags → 'on_hold' → then 'verified' or 'cancelled'
    Cancelled:        → payment amount is REVERSED from the sale's amount_paid
                         and ADDED BACK to balance_remaining

  ★ CHANGES TO createOrder():
    Current logic:
      payment_status = cash/gcash ? 'paid' : 'not_paid'

    New logic for ALL payment methods:
      if (payment_method == 'pay_later') {
          // No payment now
          amount_paid = 0
          balance_remaining = total
          payment_status = 'not_paid'
      } else {
          // Cash, GCash, COD — user enters amount
          amount_tendered = data['amount_tendered']
          if (amount_tendered >= total) {
              // Full payment
              amount_paid = total
              balance_remaining = 0
              payment_status = 'paid'
              change_amount = amount_tendered - total
          } else {
              // Partial payment
              amount_paid = amount_tendered
              balance_remaining = total - amount_tendered
              payment_status = 'partial'
              change_amount = 0
          }
          // Create Payment record for initial payment
          Payment::create(...)
      }

    - If status = 'verified' (cash):
        - Update sale: amount_paid += amount
        - Update sale: balance_remaining -= amount
        - If balance_remaining <= 0 → payment_status = 'paid', paid_at = now()
        - Else → payment_status = 'partial'
    - If status = 'pending' (gcash):
        - DON'T update sale totals yet (wait for verification)
    - Clear cache, return updated sale

  ★ NEW METHOD: verifyPayment(int $paymentId)
    - Find Payment record where status = 'pending' or 'on_hold'
    - Set status = 'verified', verified_by = auth user, verified_at = now()
    - NOW update sale: amount_paid += amount, balance_remaining -= amount
    - If balance_remaining <= 0 → payment_status = 'paid', paid_at = now()
    - Else if amount_paid > 0 → payment_status = 'partial'
    - Clear cache

  ★ NEW METHOD: holdPayment(int $paymentId, string $reason)
    - Only allowed for 'pending' payments
    - Set status = 'on_hold', hold_reason = reason
    - Sale totals NOT affected (payment not yet counted)

  ★ NEW METHOD: cancelPayment(int $paymentId, string $reason)
    - If payment was 'verified' → REVERSE:
        - sale.amount_paid -= payment.amount
        - sale.balance_remaining += payment.amount
        - Recalculate payment_status (paid/partial/not_paid)
    - If payment was 'pending' or 'on_hold' → just cancel (no reversal needed)
    - Set status = 'cancelled', cancel_reason = reason

  ★ NEW METHOD: getPaymentHistory(int $saleId)
    - Return all Payment records for a sale, ordered by paid_at
    - Include received_by and verified_by user names

  ★ NEW METHOD: getCustomerBalances(int $customerId)
    - Sales where customer_id = X and balance_remaining > 0
    - Include total outstanding across all orders
    - Include payment history per order

  ★ NEW METHOD: getAllOutstandingBalances()
    - All sales where balance_remaining > 0, grouped by customer
    - For the Payments page overview

  RULES:
  1. Partial payment orders MUST have a customer linked
     (cannot track balance for anonymous walk-ins)
  2. Full cash/gcash payments = auto verified (no admin review needed)
  3. GCash payments start as 'pending' → admin must verify
  4. When a payment is CANCELLED:
     → If it was 'verified': reverse the amount on the sale
        (amount_paid -= X, balance_remaining += X)
     → If it was 'pending' or 'on_hold': no reversal needed
        (it was never counted toward amount_paid)
  5. Payment amount must be > 0 and ≤ balance_remaining
  6. Change is only calculated when amount_tendered > total (full payment)
     For partial: change = 0 (they're not overpaying)
  7. An order can have payments with MIXED methods:
     1st: ₱500 Cash, 2nd: ₱300 GCash, 3rd: ₱200 Cash
  8. The sale's payment_method field remains the ORIGINAL method chosen
     Individual method per payment is in the payments table
  9. Voiding an order with payments:
     → Only allow void if all payments are in initial state
     → Or: void the order + mark all payments as 'cancelled'

  EDGE CASES:
  1. Customer overpays on last payment:
     → amount > balance_remaining → cap at balance, calculate change
  2. Admin cancels a verified payment on a fully-paid order:
     → payment_status reverts from 'paid' to 'partial'
     → balance_remaining recalculated
  3. Multiple pending GCash payments at same time:
     → Each tracked independently, verify individually
  4. Order total changes after partial payment:
     → Recalculate balance_remaining = new_total - amount_paid
     → Could result in overpayment if total decreased — handle as credit
  5. Offline POS records a payment:
     → Queue it, sync when online, handle conflicts
  6. Receipt reprint:
     → Always show CURRENT balance (not original)

  GCASH VERIFICATION WORKFLOW:
  -----------------------------------------------------------------------
  1. Staff at POS: Customer pays ₱500 via GCash
  2. Staff enters reference number + uploads proof screenshot
  3. Payment created with status = 'pending'
  4. Sale balance NOT updated yet (pending verification)
  5. Admin opens Payments page → sees pending payment
  6. Admin checks GCash records → money received?
     YES → Click "Verify" → sale.amount_paid += ₱500, balance -= ₱500
     NO  → Click "Hold" (reason: "No GCash received, checking...")
           → Later: "Cancel" (reason: "Payment not received")
           → OR later: "Verify" (found it, it was delayed)

  ALTERNATIVE (SIMPLER) APPROACH FOR GCASH:
  -----------------------------------------------------------------------
  If you don't want GCash payments to be pending by default:
  - All payments (Cash AND GCash) start as 'verified'
  - Admin can LATER hold/cancel if they find a problem
  - This is simpler but less "safe" — money is counted before verified
  - RECOMMENDED for small business: use this simpler approach
  - The Payments page then becomes a REVIEW tool, not a gatekeeping tool

  10. STAGGERED PAYMENT IMPLEMENTATION
================================================================================

  CONCEPT OVERVIEW:
  -----------------------------------------------------------------------
  Staggered payment is a structured payment plan where the customer and
  business agree on a FIXED SCHEDULE of payments over time. Unlike regular
  partial payments (which are ad-hoc), staggered payments have:
  
  - Predetermined payment amounts per installment
  - Fixed due dates for each installment
  - Optional down payment requirement
  - Late payment tracking and penalties (optional)
  - Automatic reminders before due dates

  Example Staggered Payment Plan:
    Order Total: ₱10,000
    Down Payment: ₱2,000 (20%)
    Remaining: ₱8,000 in 4 monthly installments of ₱2,000 each
    
    Schedule:
    • Down Payment: ₱2,000 — Due: Apr 20, 2026 (order date)
    • Installment 1: ₱2,000 — Due: May 20, 2026
    • Installment 2: ₱2,000 — Due: Jun 20, 2026
    • Installment 3: ₱2,000 — Due: Jul 20, 2026
    • Installment 4: ₱2,000 — Due: Aug 20, 2026

  KEY DIFFERENCES FROM REGULAR PARTIAL PAYMENTS:
  -----------------------------------------------------------------------
  Regular Partial:
    - Customer pays whenever they want
    - No fixed schedule
    - No due dates
    - No late fees
    - Amount per payment varies

  Staggered Payment:
    - Fixed schedule agreed upfront
    - Each installment has a due date
    - Late payment tracking
    - Optional late fees/penalties
    - Fixed amount per installment (or custom per installment)

  11. PDO (POST-DATED ORDER) — BANK PAYMENT IMPLEMENTATION
================================================================================

  CONCEPT OVERVIEW:
  -----------------------------------------------------------------------
  PDO (Post-Dated Order) is a payment method where the customer provides
  post-dated checks or commits to bank transfer on a future date. This is
  common in B2B transactions or large orders where payment is guaranteed
  but deferred to a specific future date.

  Key Characteristics:
  - Payment is PROMISED for a specific future date
  - Often backed by post-dated checks or bank commitment letter
  - Order can be fulfilled before payment (based on trust/credit terms)
  - Payment date is fixed (not a payment plan with multiple dates)
  - Can be extended/rescheduled if needed
  - May require approval from admin/manager

  Example PDO Scenarios:
  1. Corporate client orders ₱50,000 worth of goods
     - Provides post-dated check dated May 15, 2026
     - Order fulfilled immediately
     - Payment expected on May 15, 2026
  
  2. Regular customer with good credit history
     - Orders ₱20,000
     - Commits to bank transfer by April 30, 2026
     - No check provided, based on trust
     - Order fulfilled after admin approval
  
  3. Supplier payment terms (Net 30, Net 60)
     - Order placed today
     - Payment due in 30 or 60 days
     - Standard business credit terms

     - Update sale: payment_status = 'paid', pdo_status = 'fulfilled',
                    amount_paid = promised_amount, balance_remaining = 0
     - Audit log
     - Send receipt/confirmation
  
  4. extendPDODate(int $pdoId, date $newDate, string $reason)
     - Find PDO
     - If extension_count == 0: save original_date = current promised_date
     - Update: promised_date = newDate, extension_count++, notes += reason
     - If was 'overdue': change status back to 'approved'
     - Audit log
     - Send notification to customer
  
  5. markPDOAsBounced(int $pdoId, string $reason)
     - Find PDO
     - Update: status = 'bounced', bounce_reason = reason
     - Update sale: pdo_status = 'bounced'
     - Send alert to admin + customer
     - Audit log
     - (Optional) Block customer from future PDO
  
  6. cancelPDO(int $pdoId, string $reason)
     - Update: status = 'cancelled', notes += reason
     - Update sale: is_pdo = false, pdo_status = 'cancelled'
     - Decide: cancel order OR convert to different payment method
     - Audit log
  
  7. getOverduePDOs()
     - All PDOs where promised_date < today && status in ['approved', 'pending']
     - Update status to 'overdue'
     - Send overdue notifications
     - Return list for admin review
  
  8. getPDOsByStatus(string $status)
     - Filter PDOs by status
     - Include sale and customer info
     - For dashboard/reports

  BUSINESS RULES FOR PDO:
  -----------------------------------------------------------------------
  1. PDO REQUIRES customer (cannot be anonymous)
  2. PDO requires admin/manager approval before order fulfillment
  3. Minimum order amount for PDO: ₱5,000 (configurable)
  4. Maximum PDO amount per customer: based on credit rating
  5. Customer must have good payment history for PDO eligibility
  6. Post-dated checks must be dated at least 1 day in future
  7. Maximum PDO period: 90 days from order date (configurable)
  8. After 3 bounced checks, customer is blocked from PDO
  9. PDO can be extended maximum 2 times (configurable)
  10. Each extension requires admin approval + reason
  11. Overdue PDOs trigger automatic reminders (daily)
  12. Orders can be fulfilled after approval but before payment
      (risk accepted by business)
  13. If check bounces, order may be recalled or customer billed
  14. PDO terms can be customized per customer (VIP clients)
  15. All PDO actions logged in audit trail

  PDO APPROVAL CRITERIA:
  -----------------------------------------------------------------------
  Auto-approve if:
    - Customer has 5+ successful PDOs with 100% on-time payment
    - Order amount < ₱10,000
    - Customer credit rating: Excellent
  
  Require manual approval if:
    - New customer (first PDO)
    - Order amount > ₱50,000
    - Customer has 1+ late payment in history
    - Customer has outstanding balance
    - Check date > 60 days in future
  
  Auto-reject if:
    - Customer has bounced check in last 6 months
    - Customer has 3+ overdue payments
    - Customer is on PDO blocklist
    - Order amount exceeds customer credit limit

  12. INTEGRATION: STAGGERED PAYMENTS + PDO + PARTIAL PAYMENTS
================================================================================

  HOW THEY WORK TOGETHER:
  -----------------------------------------------------------------------
  The system now supports THREE payment flexibility options:
  
  1. PARTIAL PAYMENTS (Ad-hoc)
     - Customer pays whenever they want
     - No schedule, no due dates
     - Any payment method (cash, gcash, cod, pay_later)
     - Example: Pay ₱500 today, ₱300 next week, ₱200 whenever
  
  2. STAGGERED PAYMENTS (Scheduled Installments)
     - Fixed payment schedule agreed upfront
     - Each installment has due date
     - Late fees for overdue installments
     - Example: ₱2,000 down, then ₱2,000/month for 4 months
  
  3. PDO (Single Future Payment)
     - One payment on a specific future date
     - Backed by post-dated check or bank commitment
     - Order fulfilled before payment (credit terms)
     - Example: Order today, pay ₱50,000 on May 15
  
  DECISION TREE — Which to Use?
  -----------------------------------------------------------------------
  
  Customer wants to pay in parts over time?
    ├─ No fixed schedule, pay whenever → PARTIAL PAYMENTS
    │  • Most flexible
    │  • No commitment to dates
    │  • No late fees
    │
    └─ Fixed schedule with due dates → STAGGERED PAYMENTS
       • Structured payment plan
       • Late fees for missed payments
       • Better for business cash flow planning
  
  Customer wants to pay later (full amount)?
    ├─ Pay within 7-14 days, no commitment → PAY LATER (partial payment)
    │  • Simple, no paperwork
    │  • Come back and pay anytime
    │
    └─ Pay on specific future date (30-90 days) → PDO
       • Requires approval
       • Backed by check or bank commitment
       • Order fulfilled immediately
       • Good for B2B / corporate clients

  COMBINED SCENARIOS:
  -----------------------------------------------------------------------
  
  Scenario 1: PDO with Down Payment
    - Customer orders ₱50,000
    - Pays ₱10,000 down payment (cash)
    - Provides post-dated check for ₱40,000 due May 15
    - Implementation:
      • Create sale with partial payment (₱10,000 paid)
      • Create PDO for remaining ₱40,000
      • Order can be fulfilled after PDO approval
  
  Scenario 2: Staggered Payment with Missed Installment
    - Customer has payment plan: 4 x ₱2,000/month
    - Misses 2nd installment (now overdue)
    - Comes in and pays ₱1,000 (partial installment payment)
    - Implementation:
      • Record payment against installment #2
      • Installment status: 'partial' (₱1,000 of ₱2,000 paid)
      • Late fee still applies to unpaid portion
      • Customer can pay remaining ₱1,000 later
  
  Scenario 3: Convert Partial Payment to Staggered Plan
    - Customer has ₱10,000 order, paid ₱3,000 (partial)
    - Balance: ₱7,000
    - Customer requests payment plan for remaining balance
    - Implementation:
      • Create payment plan for ₱7,000 (not full order amount)
      • Generate installments for remaining balance only
      • Original ₱3,000 payment stays as-is
  
  Scenario 4: PDO Check Bounces, Convert to Staggered
    - Customer's ₱50,000 PDO check bounces
    - Customer can't pay full amount now
    - Negotiate payment plan instead
    - Implementation:
      • Mark PDO as 'bounced'
      • Create staggered payment plan for ₱50,000
      • Customer pays in installments instead
      • May add penalty/interest
  
  Scenario 5: Early Payment on Staggered Plan
    - Customer has 4 remaining installments of ₱2,000 each
    - Wants to pay off entire balance early
    - Implementation:
      • Calculate total remaining: ₱8,000
      • Record single payment for ₱8,000
      • Mark all remaining installments as 'paid'
      • Payment plan status: 'completed'
      • May offer early payment discount

  DATABASE RELATIONSHIPS SUMMARY:
  -----------------------------------------------------------------------
  
  sales (main order table)
    ├─ has many → payments (individual payment transactions)
    ├─ has one → payment_plan (if staggered payment)
    │   └─ has many → payment_plan_installments
    └─ has one → post_dated_order (if PDO)
  
  A single sale can have:
    - Multiple payments (partial payments over time)
    - One payment plan with multiple installments (staggered)
    - One PDO record (post-dated order)
    - Combination: PDO with down payment (has payments + PDO)
  
  Mutual Exclusivity:
    - A sale CANNOT have both payment_plan AND post_dated_order
    - It's either: regular partial, OR staggered plan, OR PDO
    - Exception: PDO can have initial down payment (recorded in payments)
