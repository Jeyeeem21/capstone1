POS + SHIPPING FEE + PAYMENT NEW PLAN
Date: May 1, 2026
Status: Decision-Locked Draft (ready for implementation checklist)

============================================================
1) GOAL
============================================================
Implement a dual shipping mode and role-based payment flow:

- Add ON/OFF setting for OpenRoute shipping logic.
- If OpenRoute is OFF, Admin and Super Admin can input manual price per sack
  in Delivery Address flow, and that becomes shipping fee.
- Customer Shop/POS becomes Pay Later only.
- Customer delivery shipping fee starts as NULL/pending until Admin/Super Admin
  sets it.
- Secretary POS is order-only: no shipping fee setup and no payment handling.

============================================================
2) REQUESTED BUSINESS RULES (FROM DISCUSSION)
============================================================
A. OpenRoute ON mode (current style, keep this):
- Delivery address autocomplete + distance km + estimated travel time.
- Auto shipping formula remains based on settings:
  ceil(distance_km / base_km) * rate_per_sack * total_sacks + (rate_per_km * distance_km)

B. OpenRoute OFF mode (new behavior):
- No auto distance dependency for shipping computation.
- In Admin/Super Admin POS, under Delivery Address section, show manual input:
  "Price per sack".
- Shipping fee formula in OFF mode:
  shipping_fee = price_per_sack * total_sacks
- Distance km field and auto-distance display are hidden in OFF mode.

C. Customer Shop/POS flow:
- Payment method options should be Pay Later only.
- Customer cannot set shipping fee.
- For delivery, shipping fee is NULL/pending on initial order creation.
- Customer payment starts only after Admin/Super Admin sets final shipping fee.
- No installment/staggered setup while shipping fee is pending.

D. Secretary POS flow:
- Secretary can place orders only.
- Secretary cannot set shipping fee.
- Secretary does not handle payment collection.
- Orders created by secretary should default to pay_later and unpaid status.
- For delivery orders created by secretary, shipping fee is always pending until
  Admin/Super Admin sets it.

============================================================
3) ROLE MATRIX (TARGET)
============================================================
Super Admin:
- Can toggle OpenRoute ON/OFF in Settings
- Can set manual shipping per sack when OpenRoute OFF
- Can set/update shipping fee for customer orders
- Can handle payment actions

Admin:
- Same as Super Admin for shipping and payment

Secretary:
- Can create order
- Cannot set shipping fee
- Cannot process payment

Customer:
- Can place order from Shop
- Payment method is Pay Later only
- Cannot set shipping fee

============================================================
4) SETTINGS CHANGES
============================================================
Add a new business setting key:
- shipping_openroute_enabled (boolean, default true)

Keep existing settings for ON mode:
- shipping_base_km
- shipping_rate_per_sack
- shipping_rate_per_km
- warehouse_address

Settings UI behavior:
- If shipping_openroute_enabled = true:
  show current OpenRoute shipping fields and preview.
- If shipping_openroute_enabled = false:
  show note: "Manual per-sack shipping in POS by Admin/Super Admin".

============================================================
5) DATA MODEL CHANGES
============================================================
Sales table updates (proposed):
- delivery_fee: allow NULL to represent "not set yet"
- add shipping_fee_status enum/string:
  pending | set
- add shipping_price_per_sack_override decimal nullable
- keep distance_km nullable (for optional logistics display)

Reason:
- Need explicit pending state for customer orders where shipping is not yet quoted.

============================================================
6) API/BACKEND PLAN
============================================================
A. Business settings API
- Add validation/serialization for shipping_openroute_enabled.
- Update seeder/resource/context mapping.

B. Order creation API (/sales/order)
- For customer role:
  force payment_method = pay_later
  force amount_tendered = 0
  force payment_status = not_paid
  if for_delivery: allow delivery_fee NULL and set shipping_fee_status = pending
  reject payment/installment payload when shipping_fee_status = pending

- For secretary role:
  force payment_method = pay_later
  no shipping fee write allowed from request
  no payment payload allowed from request
  if for_delivery: shipping_fee_status = pending

- For admin/super admin:
  if OpenRoute OFF and for_delivery:
    accept shipping_price_per_sack_override
    compute delivery_fee = shipping_price_per_sack_override * total_sacks
  allow setting shipping either at POS checkout or later in Admin Orders page

C. New endpoint for shipping quote update (recommended)
- PATCH /sales/{id}/shipping-fee
- Allowed roles: admin, super_admin
- Input:
  shipping_price_per_sack_override
- Server computes and stores:
  delivery_fee, total, balance_remaining, shipping_fee_status=set

D. Payment guard while shipping pending
- If shipping_fee_status = pending, block all payment recording endpoints.
- Return validation message:
  "Cannot accept payment yet. Shipping fee is still pending."
- After shipping is set, payment endpoints become available again.

============================================================
7) FRONTEND PLAN
============================================================
A. Admin Settings page
- Add OpenRoute toggle switch (ON/OFF).

B. Shared POS (Admin/Super Admin)
- Delivery Address card behavior:
  if OpenRoute ON: keep current auto km and formula breakdown.
  if OpenRoute OFF: hide distance UI and show "Price per sack" input only.
  compute shipping from sacks only.

C. Shared POS (Secretary)
- Disable payment method selection.
- Auto-set Pay Later.
- Hide/disable shipping fee inputs.

D. Customer Shop
- Payment methods list: Pay Later only.
- Show message:
  "Shipping fee will be set by admin. Payment will be available after quote."
- Submit with delivery_fee as null/pending when delivery order.

E. Admin Orders page
- Add action/button for pending-shipping orders:
  "Set Shipping Fee"
- Input per-sack rate, preview computed fee, save.

F. Payment and installment UX guard
- If shipping is pending, show badge: "Shipping Fee Pending".
- Disable payment actions and installment setup actions until shipping is set.
- Once shipping is set, enable payment and installment/staggered options based on
  final total.

============================================================
8) ORDER LIFECYCLE (TARGET)
============================================================
1. Customer places delivery order (pay_later only, shipping pending).
2. Order is created with delivery_fee = null, shipping_fee_status = pending.
3. Admin/Super Admin sets shipping per sack (at POS checkout or Orders page).
4. System computes delivery_fee and updates total.
5. Payment and installment features become enabled only after shipping is set.
6. Customer/admin can proceed with payment workflow on final total.

============================================================
9) ACCEPTANCE CRITERIA
============================================================
1. OpenRoute ON/OFF toggle is saved and affects POS behavior immediately.
2. When OFF, Admin/Super Admin sees manual "Price per sack" in delivery flow.
3. Customer Shop only allows Pay Later.
4. Customer delivery orders start with shipping fee NULL/pending.
5. Secretary cannot set shipping and cannot process payment.
6. Admin/Super Admin can set shipping later and system recalculates total correctly.
7. While shipping is pending, payment and installment actions are blocked.
8. After shipping is set, payment/installment actions use the final recalculated total.

============================================================
10) DECISIONS LOCKED (CONFIRMED MAY 1, 2026)
============================================================
1. OpenRoute OFF mode hides distance km completely.
2. Shipping fee can be set during Admin/Super Admin POS flow or later in Admin Orders page.
3. For pending shipping orders, no payment and no installment first.
4. After shipping is finalized, payment/installment can proceed on final total.
5. Secretary-created delivery orders remain pending shipping until Admin/Super Admin sets fee.
